AstraNotes UML Activity Diagram

Activity overview aligned with the implemented app behavior, requirements, architecture, finalized class diagram, updated object diagram, and updated use case diagram. The flows below focus on the real product paths and separate them into clean scenario panels to avoid crossed connectors and blocked labels.

Alignment Notes

  • This version follows the implemented session model: first launch uses passphrase setup, while later launches go directly to the workspace in a locked session and rely on step-up authentication only for secure operations.
  • Secure-note search is shown against secureTitleAlias only. The prior diagram's secure-title cache behavior was removed because the current search service does not maintain decrypted secure-title cache state.
  • Passphrase rotation is scoped to secure-note payloads and credentials. Attachment files are not re-encrypted in the current implementation, so that unsupported branch was intentionally removed.
  • Plugin management is limited to install, validate, enable, disable, and remove. The diagram does not invent plugin-execution activities that are not surfaced by the app.
  • Unsupported or undocumented behaviors from the old draft were omitted on purpose, including secure-note expiration, telemetry configuration, lock-timeout settings, voice transcription, and draft persistence when locking.
  • Fixed in this revision: The trash-restore action box previously read "Require unlocked session or step-up auth to restore secure note content," which implied a step-up authentication prompt. The actual ProtectedTrashService.restore() implementation simply throws restoreRequiresUnlockedSession — it is a hard error block, not a prompt. The box now reads "blocks with error if locked; no step-up auth triggered here."
