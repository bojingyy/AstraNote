AstraNotes UML Use Case Diagram

Use cases aligned with the current code, requirements, architecture, class diagram, and object diagram. The diagram reflects the real product scope: first-launch passphrase setup, direct workspace access on later launches, secure-note step-up authentication, local backup/import, subject groups, protected trash, attachments, and plugin management only.

Alignment Notes

  • The diagram now matches the current architecture and source code: later launches go directly to the workspace, while secure-note open/save flows reuse an in-context authentication use case.
  • Authenticate for Secure Action is included by secure-note open and secure-note save, and Use Biometrics is modeled as an extension of that authentication path rather than as a separate whole-app unlock flow.
  • Manage Installed Plugins is the base use case. Install Plugin from Local Package and Enable / Disable / Remove Plugin each optionally extend it via «extend», meaning these sub-actions are triggered only when the user explicitly chooses them — they are not mandatory inclusions. UI-triggered plugin execution is intentionally omitted because the code exposes PluginService.execute only at the service layer, not through the UI.
  • Fixed in this revision: The two arrows from Manage Installed Plugins to its sub-use-cases were previously marked «include» pointing downward from the base. They are now correctly marked «extend» with the arrowhead pointing upward (extension → base), because the install and enable/disable/remove sub-actions are optional, user-initiated extensions, not mandatory behaviors that the base always calls.
  • The external actors mirror the real boundaries in the codebase: local files for image/plugin/archive I/O, local authentication and Keychain for biometric unlock, and macOS lifecycle/power events for background/sleep lock behavior.
  • The repeated User actor glyphs are the same actor shown multiple times purely to keep associations readable and prevent line overlap.
  • Secure-note expiration, scheduling notifications, secure-title caching, transcription, lock-timeout settings, telemetry settings, and draft persistence on lock were removed because they do not exist in the current requirements, implementation, class diagram, or object diagram.
  • Title search is represented as a user use case, but it should be interpreted consistently with the docs: normal notes match plaintext titles and secure notes match secureTitleAlias only.
